# Projects

This is a list of project ideas to get you started on thinking about your project. Note that these are not complete detailed project specifications: you would need to discuss with your team and myself to come up with a clear specification. You are of course welcome to come up with your own project - please discuss with me.

1. **Sensitivity Analysis of RDMA Synchronization Primitives**
The goal of this project is to measure how the performance of underlying RDMA primitives—specifically barriers and atomics—impacts the performance of high-level concurrent data structures. You will benchmark the [LOCO (Library of Composable Objects)](https://arxiv.org/pdf/2510.10531) on a physical RDMA cluster. To isolate the impact of synchronization costs, you will emulate varying costs for the barriers and atomics and analyze the resulting throughput and tail latency of the data structures. A deep understanding of the RDMA memory consistency model is required. Please consult the OOPSLA 2024 paper: *[Semantics of Remote Direct Memory Access](https://www.soundandcomplete.org/papers/OOPSLA2024/RDMA/rdma.pdf)*.

2. **The "Consistency Tax" of CXL-Native Computing**
Simulate a CXL 3.0 system on Gem5, assume all memory is on CXL and explore the consistency cost of enforcing TSO in this model. Also explore the cost of enforcing precise exceptions. You might want to read [Imprecise Store Exceptions](https://dl.acm.org/doi/10.1145/3579371.3589087) and [The Hitchhiker’s Guide to Programming and Optimizing Cache Coherent Heterogeneous Systems CXL, NVLink-C2C, and AMD Infinity Fabric](https://arxiv.org/pdf/2411.02814).

3. There are a bunch of CXL simulators out there. This project will analyze the simulators qualitatively and quantitatively to analyze what simulator can be used in what context. For a list of CXL simulators look at [CXL-DMSim](https://arxiv.org/pdf/2411.02282). For analyzing the simulators quantitatively, run the benchmarks in this paper ([The Hitchhiker’s Guide to Programming and Optimizing Cache Coherent Heterogeneous Systems CXL, NVLink-C2C, and AMD Infinity Fabric](https://arxiv.org/pdf/2411.02814)) for validation.

4. **Porting Distributed Apps to CXL**
Distributed applications (databases, key-value stores, graph processing) are designed for a "Shared Nothing" architecture, where nodes communicate via explicit network messages. CXL changes this. With CXL 3.0, we have a "Shared Memory" architecture where nodes can directly load/store to a shared region. The goal of this project is to take a standard distributed application and port it to a CXL Pod, replacing slow network messages with fast memory loads/stores, similar to how the [Tigon](https://www.cs.utexas.edu/~witchel/pubs/huang25osdi-tigon.pdf) database was built. Since we do not have physical CXL Pods, you will use Emulation (like the Tigon authors did).

5. **Is sequential consistency (SC) actually easier to program compared to relaxed Consistency?**
Sequential Consistency, the strongest consistency model is thought be the most programmable of memory consistency models, and this fact is repeated again and again in many research papers and articles. But is this really true? Nobody knows for sure. In this project, you will investigate whether this is true by conducting a well-designed user study. You will carefully arrive a small programming problem specification, and carefully consider a relaxed memory model, depending on your hypothesis you want to establish. Then you want to analyze the results and draw insights.

6. **Specifying Heterogeneous Consistency**
Recent work has come up with [Compound Consistency Models](https://users.cs.utah.edu/~vijay/papers/pldi23.pdf) - a compositional amalgamation of memory models. That approach used an operational approach to composing memory models. Today, however, companies used the axiomatic method to expressing memory models using tools such as [Herd](https://github.com/herd/herdtools7) or [Alloy](https://github.com/NVlabs/ptxmemorymodel). The goal of the project is to express compound memory models axiomatically in a tool such as Herd. Another possibility is to extend our [work](https://github.com/goens/lost-pop-lean) and express other compound memory models such as the ARM/x86 model, ARM/PTX model etc.

7. [Hermes](https://users.cs.utah.edu/~vijay/papers/asplos20a.pdf), [Kite](https://users.cs.utah.edu/~vijay/papers/ppopp20.pdf) (and other protocols expressed in [Odyssey](https://users.cs.utah.edu/~vijay/papers/eurosys21.pdf)) are examples of replication protocols for emulating fault-tolerant shared memory. The goal of the project is to employ this fault-tolerant shared-memory abstraction to write software on top of this. For example, [lock-free data structures](https://github.com/icsa-caps/Kite), schedulers, indices etc.

8. **The Natural Memory Model that Synchronization Mandates**
Current memory models (like LKMM) are often compromises between hardware constraints and legacy C code. This project takes a "Clean Slate" approach. You will analyze the synchronization logic of a core Linux subsystem (e.g., the Scheduler or Ring Buffer) or Lock-free algorithms to derive its intrinsic ordering requirements. If we ignore x86/ARM legacy and only look at the algorithm, what are the minimal axioms required for correctness?
